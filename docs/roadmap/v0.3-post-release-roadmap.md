# PicoPet v0.3.0 发布后路线图

## 1. 当前状态

`v0.3.0` 已完成并发布为 Windows 正式版。当前产品具备：

- Windows 透明置顶桌宠窗口。
- 托盘控制：暂停、点击穿透、重置位置、缩放、开机启动、配置目录、关于、退出。
- 拖拽位置保存与重启恢复。
- 轻量行为状态机：`idle / walk / sleep / happy / dragged`。
- 行为配置：`quiet / normal / lively`，`stationary / short_range`，睡眠等待时间。
- 独立设置窗口：托盘 `设置` 打开，关闭后销毁，不常驻第二个 WebView。
- 安装版和绿色版都使用程序目录下的 `PicoPet/data/`。
- Windows release gate、installer、portable zip 和手动 QA 流程。

当前仍保持 Windows-only。macOS、Linux、AI、插件、外部皮肤包、复杂物理和账号/云同步都不进入近期 Beta 主线。

## 2. 路线原则

1. **稳定化优先**  
   `v0.3.0` 刚发布后，优先处理真实使用反馈、设置窗口细节、发布流程和维护能力，不急于扩大功能面。

2. **继续轻量**  
   不引入 React、Vue、Svelte、Tailwind、大型 UI 库、物理引擎、后台数据库或常驻 worker。设置窗口和未来工具窗口都应按需创建、关闭销毁。

3. **绿色版友好**  
   配置、日志、WebView 数据和后续用户可编辑资源继续优先放在 `PicoPet/data/` 或同级清晰目录，避免回到系统 roaming/appdata。

4. **每个阶段单独出 spec 和 plan**  
   本文只定义方向和优先级。进入具体开发前，仍按 Superpowers 流程创建独立设计规格和实现计划。

5. **先补基础能力，再扩展体验**  
   v0.3.x 先提升可维护性和可恢复性；v0.4.x 再做桌宠交互/行为增强；v0.5.x 后再考虑资源和皮肤能力。

## 3. v0.3.x：发布后稳定化与维护能力

### 目标

让 v0.3 设置窗口和 Windows Beta 发布链路更可靠、更容易排查问题，并补齐用户最可能需要的基础维护功能。

### 推荐版本拆分

#### v0.3.1：发布后修补版

优先级最高，适合在收到问题反馈后快速发布。

- 修复 v0.3.0 发布后发现的设置窗口、托盘、位置恢复、安装包或绿色版问题。
- 检查设置窗口关闭后 WebView 是否释放，必要时补充内存记录。
- 完善 release 文档中与 v0.3.0 固定版本号相关的路径，逐步改成动态版本说明。
- 更新 QA checklist，让手动验收更贴近真实用户路径。
- 保持功能面克制，不新增大功能。

验收标准：

- `pnpm release:windows:gate` 通过。
- 绿色版和安装版回归通过。
- 不新增常驻窗口或大型依赖。
- 修复项有对应测试或 QA 记录。

#### v0.3.2：设置窗口基础维护增强

在 v0.3.1 稳定后执行。

- 增加 `恢复默认设置`。
- 增加 `导出配置` 和 `导入配置`，默认仍以 `PicoPet/data/config.json` 为源。
- 增加 `打开日志文件` 或 `打开日志目录`。
- 设置窗口底部状态栏区分：保存中、保存失败、已恢复默认、导入失败。
- 保留高级用户手动编辑 JSON 的路径，但不做内置 JSON 编辑器。

验收标准：

- 配置导入会经过 Rust 侧 sanitize，不直接信任外部 JSON。
- 恢复默认不会删除 `data/EBWebView/`。
- 导入失败时不破坏现有配置。
- 设置窗口仍按需创建，关闭后销毁。

#### v0.3.3：诊断与问题反馈准备

如果后续测试人数增加，这一版优先级会上升。

- 增加轻量诊断信息导出：版本、commit、Windows 版本、WebView2 版本、配置路径、日志路径、内存摘要。
- 增加 `复制诊断信息`，便于用户反馈问题。
- release note 模板补充诊断字段。
- memory baseline 记录流程进一步标准化。

验收标准：

- 诊断信息不包含隐私路径以外的敏感内容。
- 不接入远程崩溃上报。
- 不需要管理员权限。

## 4. v0.4.x：桌宠交互与行为增强

### 目标

在基础设置和发布链路稳定后，提升桌宠的生命感，但仍保持轻量和可控。

### 推荐方向

#### v0.4.0：行为表现增强 MVP

- 增加更明确的状态表现：开心、困倦、被拖拽后的短反馈、空闲小动作。
- 补齐状态到视觉效果的映射，避免所有状态都依赖同一套 idle 表现。
- 优先使用程序化轻量效果或少量内置资源，不引入皮肤系统。
- 设置窗口中可以保留三档行为强度，但不急于增加大量参数。

验收标准：

- 默认 `quiet` 仍安静，不频繁打扰用户。
- `normal` 和 `lively` 有可感知差异。
- CPU 和内存不明显恶化。
- 行为状态变化可测试。

#### v0.4.1：`roaming` 可控漫游评估版

`roaming` 已在类型和文档中长期保留，但此前没有开放。它应该作为独立版本评估，不和其他大功能混在一起。

- 实现屏幕范围内的自主漫游。
- 支持多显示器边界处理。
- 设置窗口中把移动模式扩展为：原地、小范围、屏幕漫游。
- 提供明显的关闭路径，默认不启用 `roaming`。

验收标准：

- 桌宠不会移动到屏幕外。
- 多显示器拔插后能回到可见区域。
- 点击穿透开启时仍可通过托盘关闭或恢复。
- `roaming` 不造成持续高 CPU。

#### v0.4.2：更细的互动输入

- 双击或右键前景交互策略评估。
- 拖拽结束后的反馈动作。
- 鼠标靠近或点击后的轻量反应。
- 防误触机制，避免影响用户正常桌面操作。

验收标准：

- 点击穿透逻辑不被破坏。
- 托盘仍是最终控制入口。
- 行为增强可以通过设置关闭或降到安静。

## 5. v0.5.x：资源与皮肤基础能力

### 目标

在行为系统稳定后，再考虑用户可替换的视觉资源。第一阶段只做内置资源结构和安全加载，不做完整插件市场。

### 推荐方向

#### v0.5.0：内置多状态 spritesheet

- 从单一 idle atlas 过渡到每状态 spritesheet 或 atlas profile。
- 为 `idle / walk / sleep / happy / dragged` 提供清晰资源约定。
- 保持资源随程序打包，不开放用户导入。

验收标准：

- 缺失某状态资源时有 fallback。
- 打包体积增长可接受。
- 渲染路径仍简单，测试可覆盖 atlas/profile 解析。

#### v0.5.1：本地皮肤目录实验

- 在 `PicoPet/data/skins/` 下支持本地皮肤包。
- 定义最小 `skin.json` 元数据。
- 只支持本地文件，不做在线下载、市场、评分、更新。
- 设置窗口可选择内置皮肤或本地皮肤。

验收标准：

- 无效皮肤不会导致启动失败。
- 皮肤加载失败时回退到内置资源。
- 不执行皮肤包中的脚本。

## 6. v0.6.x 到 v1.0 前：平台化与长期能力

### 跨平台准备

跨平台不建议在 v0.4 前启动。等 Windows 主线稳定后，再拆平台层：

- 自启动接口。
- 点击穿透接口。
- 窗口置顶和透明窗口接口。
- 打开配置目录接口。
- 多屏坐标和 DPI 行为。

推荐先做平台抽象 spec，再选 macOS 或 Linux 中一个平台试点，不要同时铺开。

### 插件与扩展

插件系统不进入近期 Beta。原因是安全边界、版本兼容和用户支持成本都较高。

可以先预留：

- 资源包元数据。
- 配置迁移版本号。
- 明确的内部 command 边界。

不建议近期实现：

- 执行任意脚本。
- 在线插件市场。
- 远程自动更新插件。

### AI 与提醒

AI 聊天、提醒、语音或大模型集成属于长期方向，不应和桌宠基础体验混在同一阶段。

进入前需要先回答：

- 是否仍坚持轻量常驻。
- 是否需要联网。
- 是否需要用户账号或 API key。
- 如何处理隐私、日志和成本。

短期内只保留为长期探索项。

## 7. 推荐下一步

下一步建议进入 `v0.3.1`，目标是 **发布后修补与维护增强准备**。

推荐先做一份 `v0.3.1` 设计规格，范围控制在：

- 收集 v0.3.0 使用反馈。
- 检查 release 文档和脚本中仍然硬编码版本号的地方。
- 规划恢复默认、导出/导入配置、打开日志目录是否进入 v0.3.2。
- 补充 v0.3.0 发布后的 QA 复盘。

如果没有新的 bug 反馈，`v0.3.1` 可以只做文档、脚本和 QA 流程增强，不强行加功能。

## 8. 暂不做事项

以下事项在路线图中保留，但不建议作为下一阶段入口：

- macOS/Linux 支持。
- 外部皮肤市场。
- 插件系统。
- AI 聊天或提醒。
- 账号、云同步、在线配置。
- 自动更新器。
- 复杂物理模拟。
- 大型前端框架迁移。

这些方向都可能有价值，但它们会显著增加维护成本。当前阶段更需要一个稳定、轻量、可验证的 Windows 桌宠。
