# PicoPet v0.3 Settings Window Design

## 1. 背景

`v0.1.0` 已完成 Windows 稳定版基础能力：透明置顶桌宠窗口、托盘控制、点击穿透、拖拽位置保存、绿色版 `PicoPet/data/` 数据目录、安装包与便携包发布流程。

`v0.2.0` 和 `v0.2.1` 已完成轻量行为状态机与行为配置：桌宠支持 `idle / walk / sleep / happy / dragged` 状态，`behavior.preset` 支持 `quiet / normal / lively`，`behavior.walk_mode` 支持 `stationary / short_range`，配置保存在 `PicoPet/data/config.json`。

当前用户仍需要手动编辑 JSON 才能调整行为、窗口和启动项。`v0.3` 的重点转为基础功能补齐：提供一个轻量设置窗口，把已经稳定的配置能力暴露给普通用户，同时保持绿色版友好、低内存、Windows-only。

## 2. 目标

- 新增托盘菜单项 `设置`，打开独立设置窗口。
- 设置窗口只在用户需要时创建，关闭后销毁，不常驻第二个 WebView。
- 设置项即时保存、即时生效，不需要“应用”按钮。
- 使用类型化 Tauri commands 更新配置，不提供通用 JSON patch。
- 继续把配置保存在程序目录下的 `PicoPet/data/config.json`。
- 保持 Windows-only，不做 macOS/Linux 设置差异处理。
- 保持轻量，不引入 React/Vue/Svelte/Tailwind、状态管理库或大型 UI 组件库。
- 保留 `v0.2.1` 行为体验，不新增表情、皮肤、AI、提醒或复杂行为系统。

## 3. 非目标

- 不做设置向导、欢迎页、登录账号或云同步。
- 不做导入/导出配置。
- 不做恢复默认设置按钮；该能力等真实用户反馈后再决定。
- 不做 JSON 编辑器，只提供 `打开配置目录`。
- 不做日志中心、诊断中心或崩溃上报。
- 不做更新检查器。
- 不做跨平台设置实现。
- 不实现 `roaming` 自主漫游。
- 不暴露 `x/y` 坐标、`idle_fps`、`interactive_fps`、`always_on_top` 或内部调试字段。

## 4. 用户体验范围

设置入口：

- 托盘菜单新增 `设置`。
- 点击 `设置` 时调用 `open_settings_window`。
- 如果设置窗口不存在，则创建并显示。
- 如果设置窗口已经存在，则聚焦已有窗口，不创建重复窗口。
- 设置窗口关闭时销毁 WebView；下次打开重新读取最新配置。

窗口形态：

- 普通有标题窗口，标题为 `PicoPet 设置`。
- 建议初始尺寸约 `360 x 460`，不可过大，适合桌面工具。
- UI 风格保持安静、紧凑、工具型，不做营销式说明和大面积装饰。
- 页面分为三个区块：`行为`、`窗口`、`启动与维护`。
- 底部只显示短错误信息或当前处理中状态，不弹成功提示。

前端入口：

```text
settings.html
src/settings.ts
src/settings.css
```

设置窗口使用独立 HTML 和入口脚本，不在 `src/app.ts` 内按窗口类型分支。主桌宠入口继续由 `src/main.ts` / `src/app.ts` 管理。

## 5. 设置字段

### 5.1 行为

| UI 项 | 配置字段 | 可选值或范围 | 规则 |
| --- | --- | --- | --- |
| 行为强度 | `behavior.preset` | `quiet / normal / lively` | 默认 `quiet`，切换后立即影响行为节奏 |
| 行走模式 | `behavior.walk_mode` | `stationary / short_range` | 不展示 `roaming`，后续版本再开放 |
| 睡眠等待时间 | `behavior.sleep_after_idle_seconds` | `60-86400` 秒 | 输入值保存前 clamp 到合法范围 |

显示文案建议：

- `quiet` 显示为 `安静`
- `normal` 显示为 `标准`
- `lively` 显示为 `活跃`
- `stationary` 显示为 `原地`
- `short_range` 显示为 `小范围`

`sleep_after_idle_seconds` 使用数字输入，单位固定显示为秒。`v0.3` 不做分钟/小时换算，避免第一版设置窗口产生额外格式转换逻辑。

### 5.2 窗口

| UI 项 | 配置字段 | 可选值或范围 | 规则 |
| --- | --- | --- | --- |
| 点击穿透 | `window.click_through` | boolean | 开启后桌宠可能无法响应点击，这是预期限制 |
| 暂停动画 | `animation.paused` | boolean | 开启后暂停动画与自主行为 |
| 缩放 | `window.scale` | `0.5-2.0`，步长 `0.25` | 保存前复用 Rust 侧 clamp 规则 |
| 重置位置 | `window.x/y` | 按当前屏幕右下角计算 | 按钮触发，不暴露坐标输入 |

缩放可以使用下拉或步进控件，显示值为 `50% / 75% / 100% / 125% / 150% / 175% / 200%`。第一版不做任意小数输入。

### 5.3 启动与维护

| UI 项 | 配置字段或操作 | 规则 |
| --- | --- | --- |
| 开机启动 | `startup.launch_on_login` | 调用 Windows 自启动实现并保存配置 |
| 打开配置目录 | `PicoPet/data/` | 打开数据目录，不打开具体 JSON 文件 |

配置目录必须指向当前便携数据目录，不能退回系统 roaming/appdata 目录。

## 6. 命令与数据流

### 6.1 复用命令

`v0.3` 继续复用当前已存在的命令：

- `get_app_config`
- `set_animation_paused`
- `set_click_through`
- `set_launch_on_login`
- `set_window_scale`
- `reset_window_position`

### 6.2 新增命令

```text
open_settings_window()
set_behavior_preset(preset: String) -> AppConfig
set_walk_mode(walk_mode: String) -> AppConfig
set_sleep_after_idle_seconds(seconds: u32) -> AppConfig
open_config_dir()
```

新增命令必须遵守当前配置修复规则：

- `set_behavior_preset` 只接受 `quiet / normal / lively`。
- `set_walk_mode` 只接受 `stationary / short_range`。
- `set_sleep_after_idle_seconds` 保存前 clamp 到 `60-86400`。
- 所有修改配置的命令返回保存后的 sanitized `AppConfig`。
- 所有修改配置的命令都通过已有 `picopet://config` 事件通知主窗口刷新运行态。

### 6.3 数据流

```text
设置窗口启动
  -> get_app_config
  -> 渲染控件
  -> 用户修改控件
  -> 调用对应类型化 command
  -> Rust 校验、保存 config.json、返回 AppConfig
  -> 设置窗口用返回值刷新控件
  -> Rust 向主窗口发出 picopet://config 事件
  -> 主桌宠应用新配置
```

设置窗口不得直接写文件，也不得自己维护另一份配置缓存。前端的临时状态只用于控件 pending、错误显示和渲染。

## 7. 架构设计

### 7.1 Rust 后端

建议新增或调整边界：

```text
src-tauri/src/settings_window.rs
src-tauri/src/commands.rs
src-tauri/src/tray.rs
src-tauri/src/lib.rs
```

职责：

- `settings_window.rs`：封装设置窗口 create/focus/destroy 生命周期。
- `commands.rs`：提供行为配置、睡眠时间、配置目录和设置窗口命令。
- `tray.rs`：新增 `设置` 菜单项，只负责触发命令或窗口函数。
- `lib.rs`：注册新增 commands 和模块。

设置窗口 label 建议为 `settings`。创建时加载 `settings.html`；关闭时释放窗口引用。若主桌宠窗口不存在，设置窗口仍可打开，但依赖主窗口的命令需要返回明确错误。

### 7.2 TypeScript 前端

建议新增：

```text
src/settings.ts
src/settings.css
src/settings.test.ts
```

现有 `src/tauri/commands.ts` 扩展类型化 wrapper：

```text
openSettingsWindow()
setBehaviorPreset(preset)
setWalkMode(walkMode)
setSleepAfterIdleSeconds(seconds)
openConfigDir()
```

设置 UI 可以使用原生 HTML 控件：

- select：行为强度、行走模式、缩放。
- checkbox：点击穿透、暂停动画、开机启动。
- number input：睡眠等待时间。
- button：重置位置、打开配置目录。

无需引入前端框架。使用小型渲染函数和事件绑定即可。

### 7.3 配置事件

当前托盘和命令侧已有 `picopet://config` 配置更新事件模型。`v0.3` 需要统一所有设置命令的事件发出路径，确保：

- 主桌宠能收到设置窗口改动。
- 托盘菜单改动后设置窗口重新打开能读取最新配置。
- 设置窗口不依赖主窗口回传事件才能更新自身；它以 command 返回值为准。

## 8. 错误处理

设置窗口启动失败：

- 托盘点击 `设置` 时如果窗口创建失败，记录错误。
- 不让主桌宠退出。

读取配置失败：

- 设置窗口显示短错误信息。
- 保留 `重试` 按钮或允许刷新窗口重试。

保存失败：

- 只回滚当前控件到上一次已确认的 `AppConfig`。
- 底部显示短错误信息。
- 不弹出系统级成功/失败对话框。

依赖主窗口的命令失败：

- `set_click_through`、`set_window_scale`、`reset_window_position` 需要主窗口。
- 如果主窗口缺失，设置窗口应显示错误并保持可用。
- 行为配置、睡眠时间、开机启动、打开配置目录不依赖主窗口。

打开配置目录失败：

- 显示短错误信息。
- 不创建替代目录到 roaming。

## 9. 轻量与内存约束

- 设置窗口关闭后必须销毁，不能隐藏常驻。
- 设置窗口不启动动画循环。
- 不加载桌宠 atlas 图片。
- 不创建 worker、数据库、后台轮询或额外常驻定时器。
- 不新增大型依赖。
- 设置窗口打开期间允许增加一个 WebView 的短时内存；关闭后应释放到接近打开前状态。
- `release:windows:gate` 仍是发布前硬门槛。

## 10. 测试计划

### 10.1 Rust 单元测试

- `open_settings_window` 首次调用创建设置窗口。
- 重复调用 `open_settings_window` 聚焦已有设置窗口，不创建重复窗口。
- 设置窗口关闭后再次打开会创建新窗口。
- `set_behavior_preset` 接受 `quiet / normal / lively`，拒绝未知值。
- `set_walk_mode` 接受 `stationary / short_range`，拒绝 `roaming` 和未知值。
- `set_sleep_after_idle_seconds` clamp 到 `60-86400`。
- `open_config_dir` 使用 `PicoPet/data/`。

### 10.2 TypeScript 单元测试

- 设置窗口启动时调用 `get_app_config` 并填充所有控件。
- 行为强度 select 调用 `set_behavior_preset`。
- 行走模式 select 调用 `set_walk_mode`。
- 睡眠数字输入调用 `set_sleep_after_idle_seconds`。
- 点击穿透 checkbox 调用 `set_click_through`。
- 暂停动画 checkbox 调用 `set_animation_paused`。
- 缩放控件调用 `set_window_scale`。
- 开机启动 checkbox 调用 `set_launch_on_login`。
- 重置位置按钮调用 `reset_window_position`。
- 打开配置目录按钮调用 `open_config_dir`。
- command 返回 sanitized config 后 UI 用返回值刷新。
- command 失败时显示错误并回滚当前控件。

## 12. 验收标准

- 托盘菜单可打开 `PicoPet 设置`。
- 设置窗口重复打开不会产生多个设置窗口。
- 设置窗口关闭后销毁，下次打开重新读取配置。
- 所有 `v0.3` 暴露的设置项能即时保存并即时影响桌宠或系统设置。
- 配置仍位于 `PicoPet/data/config.json`。
- `roaming` 不在 UI 中出现，也不会被保存为有效值。
- 用户不需要手动编辑 JSON 就能完成常用基础配置。
- 设置窗口不引入大型依赖，不显著增加常驻内存。
- `pnpm release:windows:gate` 通过。

## 13. 风险与应对

| 风险 | 应对 |
| --- | --- |
| 设置窗口引入第二个 WebView 增加内存 | 按需创建，关闭销毁，不加载 atlas，不常驻 |
| 即时保存导致频繁写配置 | select/checkbox/button 直接保存，数字输入使用 change 或 blur，不按每次键入保存 |
| 前后端配置校验重复 | Rust 拥有最终校验权，前端只做控件约束和错误展示 |
| 设置窗口和托盘状态不同步 | command 返回值刷新设置窗口，Rust 配置事件刷新主窗口 |
| `roaming` 未来规划值被误开放 | UI 不展示，Rust command 不接受，文档标注后续规划 |
| `src/app.ts` 继续膨胀 | 设置窗口使用独立入口，不把设置 UI 接到主桌宠入口 |

## 14. 后续规划

`v0.3` 完成后，后续可以逐步扩展：

- `v0.3.x`：补充恢复默认、导入/导出、日志入口等维护能力。
- `v0.4.x`：在行为系统稳定后开放更多表达行为或每状态 spritesheet。
- 后续跨平台阶段：把 Windows 自启动、配置目录打开、窗口 click-through 等平台差异拆出统一接口。

这些能力不进入 `v0.3.0`，避免第一版设置窗口过重。

## 15. 规格自查

- 本规格只覆盖 `v0.3` 设置窗口 MVP，不包含新的表达行为。
- 本规格明确设置入口、窗口生命周期、字段范围、命令边界和错误处理。
- 本规格明确配置继续保存在 `PicoPet/data/`，不回到 roaming/appdata。
- 本规格明确使用独立 `settings.html` / `src/settings.ts`，不污染 `src/app.ts`。
- 本规格明确不引入前端框架、大型依赖、后台 worker 或常驻第二 WebView。
- 本规格明确 `roaming` 不在 `v0.3` UI 中开放。
- 本规格给出 Rust、TypeScript、QA 和 release gate 验收要求。
