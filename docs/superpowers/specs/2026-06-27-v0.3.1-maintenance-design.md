# PicoPet v0.3.1 Maintenance Design

## 1. 背景

`v0.3.0` 已发布为 Windows 正式版，重点完成了轻量设置窗口。当前设置窗口已经可以配置行为、窗口、启动项，并继续使用程序目录下的 `PicoPet/data/config.json`。

发布后下一步不是扩展行为或皮肤，而是补齐维护能力：用户需要能恢复默认配置、导出配置、导入配置、打开日志、生成诊断信息；开发侧也需要减少 release 文档里固定版本号带来的维护成本。

`v0.3.1` 定位为轻量维护增强版，继续坚持 Windows-only、绿色版友好、低依赖、可验证。

## 2. 目标

- 在设置窗口 `启动与维护` 区新增基础维护操作：
  - `恢复默认设置`
  - `导出配置`
  - `导入配置`
  - `打开日志文件`
  - `生成诊断信息`
- 导入导出使用固定 data 目录文件，不引入系统文件选择器。
- 导入失败时不破坏现有配置。
- 恢复默认只重置 `config.json`，不删除日志、WebView 数据或导入导出文件。
- 诊断信息先显示在设置窗口内，由用户手动复制。
- 更新 release/QA 文档，减少当前版本号硬编码。
- 保持 `pnpm check`、`pnpm artifact:windows` 和 `pnpm release:windows:gate` 作为自动验证门槛。

## 3. 非目标

- 不新增系统文件选择器。
- 不新增 Tauri clipboard/dialog 插件。
- 不新增设置子窗口、JSON 编辑器、日志中心、诊断中心或崩溃上报。
- 不做自动更新器。
- 不做 AI、提醒、语音、插件、皮肤、跨平台或新行为。
- 不暴露 `x/y` 坐标、FPS、`always_on_top` 或 `roaming`。
- 不修改绿色版 `PicoPet/data/` 目录策略。

## 4. 用户体验范围

设置窗口继续使用当前三段式结构：

- `行为`
- `窗口`
- `启动与维护`

`启动与维护` 区从当前的 `开机启动` 和 `打开配置目录` 扩展为：

```text
开机启动
打开配置目录
打开日志文件
导出配置
导入配置
恢复默认设置
生成诊断信息
```

窗口仍是普通标题窗口 `PicoPet 设置`。可以适度增加窗口高度，建议从当前约 `360 x 460` 调整到约 `360 x 560`，但不做大尺寸工具台。

### 4.1 状态反馈

所有维护按钮复用现有底部状态区域：

- 操作中：显示短文本，例如 `处理中`。
- 成功：显示短文本，例如 `已导出到 data/config.export.json`。
- 失败：显示错误文本，例如 `日志文件不存在`。

不弹系统成功/失败对话框。

### 4.2 恢复默认确认

`恢复默认设置` 不使用 modal。第一次点击只进入确认状态：

```text
再次点击确认恢复默认
```

5 秒内第二次点击才真正执行。超时后恢复普通状态。这样避免误触，也避免新增复杂弹窗状态。

### 4.3 导入导出路径

固定使用 `PicoPet/data/`：

- 导出文件：`PicoPet/data/config.export.json`
- 导入文件：`PicoPet/data/config.import.json`

用户如果想导入配置，需要先把文件放到 `config.import.json`。这比系统文件选择器更朴素，但符合当前绿色版目标，不新增权限和依赖。

### 4.4 诊断信息

`生成诊断信息` 后，在设置窗口底部显示多行只读文本，至少包含：

- 版本
- 配置目录
- 配置文件
- 日志文件

后续可以扩展 Windows 版本、WebView2 版本、commit、内存摘要，但 `v0.3.1` 不强求。当前项目已有 `get_diagnostics_info`，本版本优先把它接到 UI。

诊断按钮不直接写系统剪贴板。当前项目没有 clipboard 插件，若为了复制文字引入新插件，会扩大权限和依赖面。`v0.3.1` 先展示文本，用户可以手动复制。

## 5. Rust 命令设计

新增类型化命令，不提供通用 JSON patch：

```text
reset_config_to_defaults() -> AppConfig
export_config() -> MaintenanceFileResult
import_config() -> AppConfig
open_log_file() -> ()
```

复用已有：

```text
get_diagnostics_info() -> DiagnosticsInfo
open_config_dir() -> ()
```

### 5.1 `MaintenanceFileResult`

新增返回类型：

```rust
pub struct MaintenanceFileResult {
    pub path: String,
}
```

用于导出配置成功后返回实际文件路径。前端可以显示文件名或完整路径。

### 5.2 `reset_config_to_defaults`

规则：

- 构造 `AppConfig::default().sanitized()`。
- 写入 `PicoPet/data/config.json`。
- 更新内存中的 `AppState.config`。
- 发出 `picopet://config`，让主窗口即时刷新。
- 不删除 `data/EBWebView/`。
- 不删除 `data/picopet.log`。
- 不删除 `data/config.export.json` 或 `data/config.import.json`。

### 5.3 `export_config`

规则：

- 从内存读取当前配置。
- 经过 `sanitized()`。
- 写入 `PicoPet/data/config.export.json`。
- 返回 `MaintenanceFileResult { path }`。
- 不修改正式 `config.json`。
- 不发出 `picopet://config`，因为运行配置没有变化。

### 5.4 `import_config`

规则：

- 读取 `PicoPet/data/config.import.json`。
- 解析为 `AppConfig`。
- 经过 `sanitized()`。
- 保存到正式 `PicoPet/data/config.json`。
- 更新内存中的 `AppState.config`。
- 发出 `picopet://config`。
- 返回保存后的 `AppConfig`。

失败规则：

- `config.import.json` 不存在：返回明确错误。
- JSON 损坏：返回明确错误。
- 写入失败：返回明确错误。
- 任意失败都不能覆盖当前内存配置或正式 `config.json`。

### 5.5 `open_log_file`

规则：

- 使用 `PicoPet/data/picopet.log`。
- 如果日志存在，用 Explorer 打开或选中文件。
- 如果日志不存在，返回 `日志文件不存在` 这类明确错误。
- 不创建空日志文件。

Windows 实现建议：

```text
explorer /select,<log path>
```

如果后续发现 `/select` 在某些路径上体验不好，可以改为打开日志目录，但 `v0.3.1` 的 UI 文案仍应保持准确。

## 6. TypeScript 命令包装

在 `src/tauri/commands.ts` 新增：

```ts
export type MaintenanceFileResult = {
  path: string;
};

export function resetConfigToDefaults(): Promise<AppConfig>;
export function exportConfig(): Promise<MaintenanceFileResult>;
export function importConfig(): Promise<AppConfig>;
export function openLogFile(): Promise<void>;
```

`getDiagnosticsInfo()` 已存在，继续复用。

命名继续使用 camelCase wrapper，对应 Rust snake_case command。参数保持简单，导入导出不从前端传路径，避免前端伪造路径写出 data 目录外。

## 7. 设置窗口前端设计

`src/settings.ts` 保持单文件轻量逻辑，但需要整理维护操作，避免 `bindControls()` 继续膨胀。

建议边界：

- 保留 `renderShell()` 负责 HTML。
- 保留 `applyConfig()` 负责控件刷新。
- 新增维护操作 helper：
  - `runVoidCommand()`
  - `runMaintenanceConfigCommand()`
  - `showDiagnosticsInfo()`
  - `handleResetDefaultsClick()`

### 7.1 按钮行为

- `导出配置`：
  - 调用 `exportConfig()`。
  - 成功显示 `已导出到 data/config.export.json` 或返回路径。

- `导入配置`：
  - 调用 `importConfig()`。
  - 成功调用 `applyConfig()` 刷新所有控件。
  - 失败显示错误，控件保持旧配置。

- `恢复默认设置`：
  - 第一次点击进入 5 秒确认状态。
  - 第二次点击调用 `resetConfigToDefaults()`。
  - 成功调用 `applyConfig()`。

- `打开日志文件`：
  - 调用 `openLogFile()`。
  - 失败显示错误。

- `生成诊断信息`：
  - 调用 `getDiagnosticsInfo()`。
  - 在底部或维护区下方显示只读多行文本。

### 7.2 样式

`src/settings.css` 继续保持紧凑工具型：

- 维护按钮使用 grid，避免一列太长。
- 多行诊断文本使用 `pre` 或 readonly `textarea`。
- 文本不能溢出窗口宽度。
- 不使用卡片嵌套卡片。
- 不引入图标库。

## 8. 发布流程更新

当前 `scripts/assert-windows-artifacts.ps1` 和 `scripts/package-windows-portable.ps1` 已按 `package.json` 动态读取版本，保留。

`v0.3.1` 需要调整：

- README 中固定的 `PicoPet_0.3.0_x64-portable.zip` 描述改为“按当前 package version 生成”。
- `docs/release/windows-release-process.md` 中固定 `PicoPet_0.3.0...` 路径改为动态版本说明。
- `scripts/windows-release-gate.ps1` manual reminder 指向新的 `docs/qa/windows-v0.3.1-maintenance-checklist.md`。
- 新增 `docs/qa/windows-v0.3.1-maintenance-checklist.md`。

版本号提升到 `0.3.1` 不包含在功能实现第一阶段，建议在 release-prep 阶段单独提交，沿用 v0.3.0 的发布节奏。

## 9. QA 清单

新增 `docs/qa/windows-v0.3.1-maintenance-checklist.md`，至少覆盖：

- 启动设置窗口。
- 点击 `打开配置目录`，确认打开 `PicoPet/data/`。
- 点击 `打开日志文件`，确认能定位 `picopet.log`。
- 删除或重命名日志后点击 `打开日志文件`，确认显示错误且程序不退出。
- 修改任一设置后导出配置，确认生成 `data/config.export.json`。
- 把有效配置放到 `data/config.import.json` 后导入，确认 UI、`config.json` 和主窗口运行态刷新。
- 放入坏 JSON 后导入，确认 `config.json` 保持不变。
- 恢复默认需要二次点击确认。
- 恢复默认后 `config.json` 恢复默认，但 `data/EBWebView/` 和日志不被删除。
- 生成诊断信息，确认显示版本、配置目录、配置文件、日志文件。
- 绿色版和安装版目录结构仍保持 `PicoPet/data/`。

## 10. 测试计划

### 10.1 Rust 单元测试

- `ConfigStore` 支持导入有效配置并 sanitize。
- 导入坏 JSON 不覆盖原配置。
- 导入缺失文件返回错误。
- 导出配置写到 `config.export.json`。
- 恢复默认写入 default sanitized config。
- 日志文件路径使用 `PicoPet/data/picopet.log`。
- 诊断信息仍包含 portable paths。

### 10.2 TypeScript 单元测试

- 设置窗口渲染新增维护按钮。
- `恢复默认设置` 第一次点击不调用 command。
- `恢复默认设置` 第二次点击调用 `resetConfigToDefaults()` 并刷新控件。
- `导出配置` 成功显示返回路径。
- `导入配置` 成功刷新控件。
- `导入配置` 失败显示错误且不改控件。
- `打开日志文件` 失败显示错误。
- `生成诊断信息` 显示多行信息。

### 10.3 自动验证

开发阶段：

```powershell
pnpm check
pnpm artifact:windows
```

发布前：

```powershell
pnpm release:windows:gate
```

## 11. 验收标准

- 设置窗口提供恢复默认、导出、导入、打开日志、生成诊断信息。
- 导入导出固定使用 `PicoPet/data/config.import.json` 和 `PicoPet/data/config.export.json`。
- 导入失败不破坏当前配置。
- 恢复默认不会删除日志或 WebView 数据。
- 诊断信息可以在设置窗口内查看和手动复制。
- 不新增大型依赖或 Tauri 插件。
- 不新增常驻窗口、worker、数据库或后台轮询。
- 绿色版和安装版仍使用程序目录下的 `PicoPet/data/`。
- `pnpm check` 和 `pnpm artifact:windows` 通过。
- release 前 `pnpm release:windows:gate` 通过。

## 12. 风险与应对

| 风险 | 应对 |
| --- | --- |
| 用户期待系统文件选择器 | 文档和状态文本明确固定导入导出文件名，后续版本再评估 dialog 插件 |
| 恢复默认误触 | 使用 5 秒二次点击确认 |
| 导入坏配置破坏当前配置 | 先读取和解析到临时 config，sanitize 成功后再保存和替换内存 |
| 诊断信息泄露过多 | v0.3.1 只显示版本和本地路径，不采集远程信息 |
| 设置窗口变高变复杂 | 维护按钮使用紧凑 grid，不新增子窗口 |
| release 文档版本号再次漂移 | 当前 release 文档描述改为基于 package version |

## 13. 规格自查

- 本规格只覆盖 `v0.3.1` 维护能力，不包含行为增强、皮肤、插件或跨平台。
- 本规格明确导入导出固定使用 `PicoPet/data/` 下的文件，不新增文件选择器。
- 本规格明确恢复默认、导入、导出、日志、诊断的命令边界和失败规则。
- 本规格明确不新增 Tauri 插件或大型依赖。
- 本规格明确测试、QA 和 release gate 要求。
